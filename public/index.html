Overview

Supabase Project Pinger keeps your Supabase projects exercised on a schedule so they stay responsive and easier to debug at a glance.

Traffic is driven by GitHub Actions (every 15 minutes) and this dashboard, which calls your projects the same way the automation does.

What you get

The dashboard and workflow provide:

  • Scheduled pings across many Supabase projects
  • Live status view and optional manual ping (rate-limited)
  • Per-project REST and database check results
  • GitHub Actions history and ping-results artifacts in the GitHub tab
How it runs

A workflow on GitHub installs dependencies, loads your configuration from secrets, runs the Node pinger, uploads ping-results.json as an artifact, and can be triggered on a cron, on push, or by hand.

Each project is hit on the Supabase REST root and, when available, an optional rpc/ping call so you see both API reachability and database activity in one pass.

Reading status checks

REST layer. A 200 from the REST entrypoint means PostgREST answered for that project with the credentials you configured—your edge URL and API surface are up.

Database signal. A 200 on the optional RPC step means that call completed; if the RPC is missing, the UI may show REST only for that run.

When something fails, the card shows an error string; frequent causes include:

  • Cold or paused project waking up
  • Network or timeout issues
  • API key or permission mismatch
  • RPC not deployed or not granted to the key you use
Why use a pinger

Small demos and side projects often sit idle; periodic traffic reduces “first load after silence” surprises, gives you an early warning when keys or endpoints break, and pairs with Row Level Security so publishable-style keys stay appropriate for what the checks touch.

Technology

Primary stack:

  • Node.js + Axios for HTTP
  • GitHub Actions for scheduling and artifacts
  • Vercel serverless /api/ping and /api/github for this UI
  • Supabase REST (and optional RPC) for each project
